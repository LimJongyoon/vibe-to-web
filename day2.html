한 세션 = 한 피처 원칙

다른 거 다 까먹어도 이거 하나만

이론

Claude는 모든 대화를 머릿속에 들고 다닙니다. 대화가 길어질수록 더 무거워지고 결국 헷갈리기 시작합니다. 신선한 컨텍스트가 부풀어진 컨텍스트보다 항상 낫습니다.

❌ 자주 하는 실수
> 자기소개 페이지 만들어줘
> 폰트 바꿔줘
> 다크모드 추가해줘
> 취미 섹션도 넣어줘
> 사진 갤러리 만들어줘
> ... (5시간 후)
> 처음에 만든 메인 제목 색깔 검정으로

→ 이 시점에서 Claude는 헛소리를 시작합니다.

✅ 올바른 워크플로우
① 작업 1 시작
   ↓
② 작업 1 완료
   ↓
③ /clear  ← 여기 중요
   ↓
④ 작업 2 시작
   ↓
⑤ 작업 2 완료
   ↓
⑥ /clear → 반복
실습 
[작업 1] 자기소개 페이지 만들기
   → 다 만들고 결과 확인
   → /clear ✨

[작업 2] 취미 갤러리 페이지 추가
   → 다 만들고 결과 확인
   → /clear ✨
💡
/clear는 파일을 지우지 않습니다. 대화 기억만 지우고, 진짜 파일은 그대로 남습니다. 안심하고 자주 쓰세요.

토큰 사용량 확인 — /context

> /context

바(bar) 형태로 시각화됩니다. 80% 이상이면 정리 타이밍.

명령어차이점언제 쓰나
/clear완전 초기화. 대화 다 잊음새 작업 시작 시 (가장 많이 씀)
/compact요약해서 압축. 맥락은 유지같은 작업 계속하되 공간 확보
확인
  • "한 세션 = 한 피처" 원칙을 외웠다
  • /clear가 파일을 안 지운다는 걸 안다
  • /context로 토큰 사용량 확인했다
  • /clear vs /compact 차이를 안다

Plan Mode를 습관처럼

Day 1에서 잠깐 다뤘는데, 오늘 본격적으로 씁니다

이론
모드의미
Plan Mode계획만 세움 (실행 X) — 잘못된 방향을 30초에 잡을 수 있음
Accept Mode계획 + 실행까지 함 (기본)
✅ Plan Mode가 필요한 경우
  • 새 기능을 통째로 추가할 때 (갤러리, 게시판 등)
  • 여러 파일을 동시에 건드리는 작업
  • 30분 이상 걸릴 것 같은 작업
그냥 Accept Mode로 가도 되는 경우
  • 색깔, 폰트 같은 단순 수정
  • 에러 한두 줄 고치기
  • 파일 한 곳만 살짝 만지기
실습

Plan Mode 실전 사용법

  1. 모드 전환
    [Shift + Tab 누름]
  2. 큰 작업 설명
    > 내 자기소개 페이지에 다크/라이트 모드 토글 기능을 추가하고 싶어.
  오른쪽 상단에 버튼 하나 두고, 클릭하면 전체 색상이 부드럽게 전환.
  새로고침해도 마지막 선택이 유지되면 좋겠어.
  3. Claude의 계획 받기
    다음과 같이 진행하려고 합니다:
1. 색상 변수화 — :root와 .light-mode 두 군데에 정의
2. 토글 버튼 추가 — 우상단 fixed, 클릭 시 body에 .light-mode 토글
3. 상태 저장 — localStorage
4. 전환 애니메이션 — CSS transition 0.3s

이대로 진행할까요?
  4. 검토 후 결정
    마음에 들면
    [Shift + Tab → Accept Mode]
> 좋아 진행해줘
    수정 요청
    > 좋은데, localStorage는 빼고
  body 클래스만으로 가자.
💡
습관 팁: 새 채팅 시작할 때 첫 명령어로 Shift+Tab을 누르세요. 큰 작업이면 그대로, 작은 거면 다시 눌러서 Accept Mode로.
확인
  • Plan Mode와 Accept Mode 전환 가능
  • Plan Mode를 써야 할 상황을 안다
  • 실제로 Plan을 받아보고 계획을 읽고 수정 요청해봤다

/memory로 나를 기억시키기

Claude한테 나를 기억시켜봅시다

실습

기억 저장하기

> 앞으로 모든 코드에 한국어 주석을 달아줘. 기억해.
> 내 이름은 임종윤이고, 서강대 아트앤테크놀로지 학생이야. 기억해줘.
💡
"기억해줘"라는 표현이 트리거예요. 이 단어를 쓰면 Claude가 알아서 저장합니다.

기억된 내용 확인하기

> /memory
# Memory
- 사용자 이름: 임종윤
- 소속: 서강대 아트앤테크놀로지
- 선호 폰트: Pretendard
- 선호 컬러: 다크 모드, 오렌지 액센트
- 코드 주석은 항상 한국어

기억 삭제 / 수정

> 내가 Pretendard 폰트 좋아한다는 거 기억에서 지워줘.
  이제 Noto Sans KR을 더 좋아해.

효과 검증 — 새 세션에서 확인

/clear

💡

멀티 파일로 쪼개기

HTML 하나에 CSS, JS가 다 들어있는 구조를 깔끔하게 정리합니다

이론
❌ 단일 파일의 문제
index.html (300줄)
about.html (300줄)
projects.html (300줄)

색깔 하나 바꾸려면 세 파일 다 수정.

✅ 멀티 파일의 해결
index.html
about.html
projects.html
assets/
  style.css   ← 디자인은 한 곳에
  script.js   ← 동작은 한 곳에

CSS 한 번 고치면 모든 페이지에 반영.

실습

CSS 분리시키기

> 지금 index.html에 들어있는 CSS를 별도 파일로 분리해줘.

요구사항:
- assets/ 폴더 새로 만들고 그 안에 style.css
- index.html에는 <link>로 연결
- CSS 변수(:root)로 색상을 관리하도록 정리
- 한국어 주석 추가

새 페이지 추가하기

> portfolio.html 페이지를 새로 만들어줘.
- 기존 style.css를 동일하게 사용
- 헤더에는 index.html과 동일한 네비게이션
- 본문에는 3개의 프로젝트 카드 (hover 시 살짝 떠오르는 효과)

JavaScript도 분리

> 다크/라이트 토글 기능을 추가하고 싶어.
- assets/script.js 파일에 분리
- 모든 페이지에서 동작하도록
확인

결과 폴더 구조

my-first-web/
├── index.html
├── portfolio.html
├── about.html
└── assets/
    ├── style.css
    └── script.js
  • CSS를 별도 파일로 분리해봤다
  • 새 페이지를 추가하고 같은 CSS를 재사용했다
  • JS도 별도 파일로 분리해봤다

폴더별 CLAUDE.md 전략

토큰 절약 + 정확도 향상

이론

CLAUDE.md가 500줄이 되면? 매 세션마다 그 500줄을 다 읽습니다. 토큰이 줄줄 새요.

❌ 내용 직접 작성
# CLAUDE.md
## API 엔드포인트
- POST /api/auth/login - 로그인 처리...
- GET /api/users/:id - 유저 조회...
- ... (50줄 더)
✅ 참조 링크만
# CLAUDE.md
## 프로젝트 문서
- API 스펙: @docs/api-spec.md
- DB 스키마: @docs/db-schema.md

@파일경로: 관련 작업이 생겼을 때만 읽음.

폴더별 CLAUDE.md

my-first-web/
├── CLAUDE.md             ← 프로젝트 전체 규칙
├── assets/
│   ├── CLAUDE.md         ← assets 폴더 규칙
│   └── style.css
└── pages/
    ├── CLAUDE.md         ← pages 폴더 규칙
    └── about.html

Claude가 pages/about.html을 수정할 때 루트 CLAUDE.md + pages/CLAUDE.md도 같이 읽습니다. 해당 폴더 작업할 때만 그 규칙 적용.

실습 
> CLAUDE.md가 너무 길어졌어. 정리해줘.

방향:
- 루트 CLAUDE.md에는 핵심 원칙과 참조 링크만 (50줄 이내)
- 상세 내용은 docs/ 폴더로 분리
- @ 표기법으로 참조 연결
⚠️
비기너 프로젝트는 단일 CLAUDE.md로 충분합니다. 너무 일찍 최적화하지 마세요. 200줄을 넘을 때 쪼갤 시점입니다.
확인
  • @파일경로 참조 표기법을 안다
  • 폴더별 CLAUDE.md 개념을 이해했다
  • 아직 안 쪼개도 된다는 것도 안다 (조기 최적화 X)

📖 이 문서를 보는 법

D, E 파트는 처음 보면 어렵게 느껴질 수 있는 개념(Git, Skills, Sub-Agent, MCP, Hooks)이 몰려 있습니다. 그래서 각 항목을 이렇게 나눴습니다.

  • 💚 꼭 필요한 것 — 여기까지만 따라 해도 오늘 실습을 끝까지 완주할 수 있습니다.
  • 📎 선택 심화 — 시간이 남거나 더 깊이 알고 싶을 때만 열어보면 됩니다. 안 열어봐도 오늘 실습에는 지장이 없습니다.

각 단계는 "왜 하는지 → 그대로 따라하기 → 확인" 순서로 되어 있습니다. 중간에 막히면 바로 위 단계로 돌아가서 다시 확인하세요.

Git에 올리기

코드의 시간 여행 머신

왜 하는지

지금까지 만든 파일은 내 컴퓨터에만 있습니다. 컴퓨터가 고장 나거나 실수로 파일을 지우면 다 사라집니다. Git은 "이 시점의 파일 상태를 저장해두는 기능"이고, GitHub은 그렇게 저장한 걸 인터넷에 보관해주는 사이트입니다.

비유하자면, Git으로 저장하는 행위(커밋)는 게임의 "세이브 포인트"를 만드는 것과 같습니다. 나중에 뭔가 잘못돼도 그 세이브 포인트로 돌아갈 수 있습니다. GitHub은 그 세이브 파일을 클라우드에 백업해두는 것입니다.

앞으로 나올 용어 3개만 먼저 알아두면 됩니다.

저장소(repository) — 내 프로젝트 폴더를 Git이 추적하는 상태로 만든 것
커밋(commit) — 지금 이 순간의 파일 상태를 저장하는 것 (세이브 포인트)
푸시(push) — 내 컴퓨터에 저장한 커밋을 GitHub(인터넷)로 올려보내는 것
💡
오늘은 GitHub에 코드를 안전하게 올려두는 것까지만 합니다. 실제로 인터넷에 배포(Vercel)하는 건 내일 Day 3에서 합니다.
그대로 따라하기
  1. GitHub 계정 만들기
    1. 브라우저에서 github.com 접속
    2. 오른쪽 위 Sign up 클릭
    3. 이메일, 비밀번호, 아이디 입력 (아이디는 나중에 github.com/내아이디로 쓰이니 신중하게)
    4. 이메일 인증 → 무료 플랜 선택
  2. GitHub Desktop 설치 (터미널 명령어 없이 마우스로만 진행)
    1. desktop.github.com 접속 → 다운로드 → 설치
    2. 실행 후 GitHub 계정으로 로그인
  3. 내 프로젝트 폴더를 저장소로 등록
    1. GitHub Desktop에서 File → Add Local Repository
    2. 지금 작업 중인 폴더(my-first-web 등) 선택
    3. "이 폴더는 아직 Git 저장소가 아닙니다"라는 메시지가 뜨면 Create a Repository 클릭
  4. 첫 커밋 만들기
    1. 왼쪽에 변경된 파일 목록이 보입니다
    2. 아래 Summary 칸에 작업 완성 같은 짧은 설명 입력
    3. Commit to main 클릭 → 이게 세이브 포인트 1개가 만들어진 것입니다
  5. GitHub에 올리기(푸시)
    1. 위쪽 Publish repository 클릭
    2. "Keep this code private" 체크 해제 (공개로 올려야 나중에 배포 링크 공유 가능)
    3. Publish 클릭
  6. 확인

    브라우저에서 github.com/내아이디 접속 → 방금 올린 저장소가 보이면 성공입니다.

이후 수정할 때마다

파일을 고친 뒤 GitHub Desktop을 열면 왼쪽에 바뀐 부분이 자동으로 표시됩니다. Summary 입력 → Commit to main → 위쪽 Push origin 버튼만 누르면 됩니다. 이 세 단계(수정 → 커밋 → 푸시)가 앞으로 계속 반복되는 전체 흐름입니다.

⚠️
비밀번호나 API 키가 들어있는 파일은 절대 올리지 마세요. Claude에게 ".gitignore 파일 만들어줘, 비밀번호 파일은 Git에 안 올라가게 해줘"라고 요청하면 자동으로 처리해줍니다.
확인
  • GitHub 계정을 만들었다
  • GitHub Desktop으로 내 폴더를 저장소로 등록했다
  • 커밋 → 푸시를 한 번 해봤다
  • 브라우저에서 내 코드가 GitHub에 올라간 걸 확인했다
📎 선택 심화 — 터미널로 하는 방법 (마우스 방식이 익숙해진 뒤에) 
# 최초 한 번만 설정
git config --global user.name "본인이름"

# 새 저장소 만들어서 올리기

git add .

Skills & 슬래시 커맨드 — 반복 작업 자동화

반복 작업을 AI에게 위임하는 두 가지 방법

왜 하는지

같은 요청을 매번 길게 타이핑하고 계신가요? 예를 들어 "다크모드로, Pretendard 폰트로, 반응형으로 만들어줘"를 새 페이지 만들 때마다 반복 입력한다면, 그걸 파일 하나로 저장해두고 재사용하는 기능이 Skills와 슬래시 커맨드입니다.

둘의 차이는 딱 하나입니다.

Skill슬래시 커맨드
실행 방법Claude가 상황 보고 알아서 실행내가 /이름을 직접 입력해야 실행
비유상황 되면 자동으로 켜지는 센서등내가 눌러야 켜지는 스위치
그대로 따라하기

슬래시 커맨드부터 (더 쉬운 쪽)

  1. 폴더와 파일 만들기

    프로젝트 폴더 안에 .claude/commands/commit.md 파일을 만듭니다. (Claude에게 ".claude/commands/commit.md 파일 만들어줘"라고 시켜도 됩니다.)

  2. 내용 채우기
    1. git status/git diff로 변경사항 확인
2. 변경 내용을 요약한 한국어 커밋 메시지 초안 작성
3. 사용자에게 확인 요청
4. 승인되면 git add -A && git commit -m ... 실행
  3. 실행해보기
    > /commit

    입력하면 Claude가 위 4단계를 순서대로 실행합니다. 방금 14단계에서 마우스로 하던 커밋 작업을 이제 이 한 줄로 대체할 수 있습니다.

이어서 Skill (자동 트리거)

  1. 폴더와 파일 만들기

    .claude/skills/새-페이지-만들기/SKILL.md 파일을 만듭니다.

  2. 맨 위에 이렇게 채우기 (이 형식이 고정입니다)
    ---
name: 새-페이지-만들기
description: "새 HTML 페이지를 만들 때 자동 트리거.
  기존 style.css와 헤더 네비게이션을 그대로 재사용."
---

## 절차
1. 기존 index.html의 헤더 구조를 그대로 복사
2. assets/style.css를 <link>로 연결
3. 본문만 요청받은 내용으로 채움

    description 줄이 핵심입니다. Claude는 이 한 줄을 보고 "지금 이 Skill을 써야 하나?"를 판단합니다. 그래서 "언제 쓰는지"가 최대한 구체적으로 적혀 있어야 합니다.

  3. 확인해보기

    새 페이지를 만들어달라고 평소처럼 요청해보세요.

    > portfolio.html 페이지 만들어줘. 프로젝트 카드 3개 넣어서.

    명령어를 직접 안 쳤는데도, 방금 만든 Skill의 description과 맞아떨어지면 Claude가 알아서 그 규칙을 적용합니다.

확인
  • .claude/commands/commit.md를 만들고 /commit으로 실행해봤다
  • .claude/skills/ 안에 Skill 하나를 만들어봤다
  • 명령어 없이 요청했는데 Skill이 자동으로 적용되는 걸 확인했다
📎 선택 심화 — 기성 Skill 설치해서 쓰기

직접 만들지 않고 남이 만든 Skill을 설치해서 쓸 수도 있습니다.

npx skills find "design"
npx skills add anthropics/skills@frontend-design

설치 후 "frontend-design Skill로 내 페이지 다시 디자인해줘"라고 요청하면, 방금 직접 만든 Skill과 똑같은 방식으로 자동 적용됩니다.

Sub-Agent — 도우미 하나 더 부르기

메인 Claude 안에서 도우미를 하나 더 띄우는 것

왜 하는지

지금까지는 Claude 하나와만 대화했습니다. Sub-Agent는 "이 작업만 전담하는 도우미를 하나 더 불러서 시키는 것"입니다. 예를 들어 코드 검토처럼 결과물이 길게 나오는 작업을 Sub-Agent에게 맡기면, 그 긴 결과가 메인 대화창을 어지럽히지 않고 요약만 돌아옵니다.

처음에는 "이미 만들어져 있는 Sub-Agent를 불러 쓰는 법"만 알면 충분합니다. 직접 만드는 건 선택 심화로 뒤에 있습니다.

그대로 따라하기

그냥 자연어로 요청하기 (가장 쉬움)

> 서브에이전트로 최근 변경사항 코드 리뷰해줘

"서브에이전트로"라는 말만 넣어도 충분합니다. 어떤 도우미를 쓸지는 Claude가 알아서 고릅니다.

기본으로 내장된 도우미 5종 (이미 있으니 만들 필요 없음)

이름용도
Explore코드/파일 탐색, 구조 파악
PlanPlan Mode에서 계획 세울 때 조사
General-purpose탐색+수정이 같이 필요한 복잡한 작업
Bash터미널 명령만 따로 실행
Claude Code GuideClaude Code 사용법 질문 답변

확인해보기

> 서브에이전트로 이 프로젝트 폴더 구조 파악해서 알려줘

실행하면 잠깐 별도로 작업하는 표시가 뜨고, 끝나면 요약된 결과만 메인 대화에 나타납니다. 그게 Sub-Agent가 작동한 것입니다.

확인
  • "서브에이전트로 ~해줘" 형태로 한 번 요청해봤다
  • 결과가 요약되어서 돌아오는 걸 확인했다
📎 선택 심화 — 내 전용 Sub-Agent 직접 만들기

직접 파일을 쓰는 대신, 그냥 Claude에게 만들어달라고 요청하는 게 제일 쉽습니다.

> code-reviewer라는 서브에이전트를 만들어줘.
  코드 품질/보안 검토 역할,
  Read/Grep/Glob/Bash 도구만 쓰게 해줘.

Claude가 .claude/agents/code-reviewer.md 파일을 아래와 같은 형식으로 자동 생성해줍니다. 직접 쓸 경우 형식은 이렇습니다.

---
name: code-reviewer
description: "코드 리뷰 전문가. 품질, 보안, 모범 사례를 검토."
tools: Read, Grep, Glob, Bash
model: sonnet
---

호출되면:
1. git diff로 최근 변경 확인
2. 수정된 파일에 집중하여 리뷰

만든 뒤에는 @code-reviewer처럼 이름 앞에 @을 붙여서 확실하게 지정 호출할 수도 있습니다.

⚠️
예전 버전에서는 /agents로 마법사 UI가 떴지만, 최신 버전에서는 파일 기반으로만 만듭니다. /agents는 지금 실행 중인 서브에이전트를 보는 모니터링 화면입니다.

MCP — 외부 서비스와 연결하기

Claude가 외부 도구를 표준화된 방식으로 쓰게 해주는 프로토콜

왜 하는지

Claude가 내 컴퓨터 파일뿐 아니라 Slack, Notion 같은 외부 서비스도 다룰 수 있게 해주는 연결 방식입니다. "코드를 직접 짜지 않아도 되는 플러그인을 꽂는 것"이라고 생각하면 됩니다.

오늘은 이미 존재하는 서버를 연결해서 써보는 것까지만 합니다. 커스텀 서버를 직접 만드는 건 나중에 필요할 때 해도 늦지 않습니다.

그대로 따라하기
  1. 연결하기
    # 내 컴퓨터에서 실행되는 도구를 연결할 때
claude mcp add <이름> -- npx -y <패키지이름>

# 인터넷 너머 서버에 연결할 때
claude mcp add --transport http <이름> <서버주소>

    <이름>, <패키지이름>, <서버주소>는 실제 값으로 바꿔서 입력합니다. 예를 들어 파일시스템 도구를 연결한다면 이렇게 됩니다.

    claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem
  2. 잘 연결됐는지 확인
    claude mcp list

    방금 연결한 이름이 목록에 뜨면 성공입니다.

  3. 사용해보기

    연결만 하면 별도로 부르지 않아도, 관련된 요청을 할 때 Claude가 알아서 그 도구를 씁니다.

확인
  • claude mcp add로 서버 하나를 연결해봤다
  • claude mcp list로 연결 상태를 확인했다
📎 선택 심화 — MCP 대신 간단한 Bash 스크립트로 충분한 경우

작업이 단순하면 MCP 연결 없이 로컬 스크립트만으로 충분합니다. 연결 과정도 없고 대화창 공간도 아낄 수 있습니다.

#!/bin/bash
# scripts/check-db.sh
psql -h localhost -U dev -d mydb -c "SELECT count(*) FROM users;"
> check-db.sh 실행해줘

MCP는 "매 대화마다 도구 설명이 자동으로 포함되는" 방식이라, 정말 반복적으로 외부 서비스와 연동해야 할 때만 쓰는 게 맞습니다. 간단한 일회성 작업은 그냥 Bash가 더 가볍습니다.

커스텀 MCP 서버가 필요해지면, 이렇게 요청만 하면 Claude가 코드 작성부터 등록까지 도와줍니다.

> Gemini API를 사용하는 plan-review MCP 서버를 만들어줘.
  review_plan 도구 하나만 있으면 돼.

Hooks — 자동화 엔진

특정 상황이 되면 자동으로 실행되는 규칙

왜 하는지

"파일을 저장할 때마다 자동으로 정리해줘" 같은, 특정 상황이 되면 자동으로 실행되는 규칙입니다. 오늘은 가장 흔히 쓰는 예시 하나(작업 끝나면 알림 보내기)만 직접 만들어봅니다.

그대로 따라하기 
> 알림 Hook 만들어줘.
  작업이 끝나면 소리랑 알림 띄워줘.

이렇게 요청하면 Claude가 알아서 설정 파일(settings.json)에 필요한 내용을 추가해줍니다. 직접 파일을 열어서 수정할 필요는 없습니다.

실행해서 확인하기

아무 작업이나 하나 시켜본 뒤, 끝났을 때 알림이 뜨는지 확인합니다.

⚠️
Hook이 실행되는 동안 Claude는 멈춰서 기다립니다. 만든 뒤 반응이 없다면 "Hook이 멈춰있는 것 같아, timeout 설정해줘"라고 요청하세요.
확인
  • 알림 Hook을 만들어봤다
  • 작업이 끝났을 때 알림이 뜨는 걸 확인했다
📎 선택 심화 — Hook이 반응하는 4가지 시점

지금 만든 알림 Hook은 아래 표에서 Stop 시점에 해당합니다. 필요할 때 아래 표를 참고해서 다른 시점의 Hook도 만들 수 있습니다.

시점언제예시
PreToolUse도구를 실행하기 직전위험한 명령어 실행 전에 승인 요청
PostToolUse도구 실행 직후코드 저장 직후 자동 포맷팅
Notification사용자 응답을 기다릴 때알림 전송
StopClaude의 턴이 끝났을 때완료 알림, 최종 정리

Day 2 종합 실습

오늘 배운 걸 한 번에 이어서 해보기

왜 하는지

오늘 배운 걸 다음 순서로 한 번에 이어서 해봅니다. 순서만 따라가면 됩니다.

그대로 따라하기
  1. Plan Mode로 설계

    Shift + Tab → "index.html을 멀티 파일 구조로 분리하고 싶다" 요청 → 계획 확인 후 Shift + Tab으로 Accept Mode

  2. 분리 실행

    CSS는 assets/style.css, JS는 assets/script.js로 분리 요청 → 결과 확인

  3. 슬래시 커맨드 만들기

    15단계에서 만든 /commit 준비해두기

  4. 새 페이지 추가

    15단계에서 만든 Skill 또는 기성 frontend-design Skill로 portfolio.html 추가

  5. Sub-Agent로 검토

    "서브에이전트로 지금까지 만든 코드 리뷰해줘" 요청

  6. GitHub에 올리기

    /commit 실행 → GitHub Desktop에서 Push origin

확인
  • 여러 페이지가 서로 링크로 연결되어 있다
  • CSS/JS가 별도 파일로 분리되어 있다
  • Skill 또는 슬래시 커맨드를 하나 이상 실제로 써봤다
  • Sub-Agent 리뷰를 받아봤다
  • GitHub에 코드가 올라가 있는 걸 브라우저로 확인했다
