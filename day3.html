상태줄(Status Line) 꾸미기

가장 쉬운 커스터마이즈 — 터미널 하단에 내 정보를 항상 표시

이론

Claude Code 창 하단의 상태줄(Status Line)은 현재 모델·작업 폴더·컨텍스트 사용량 같은 정보를 항상 보여주는 영역입니다. /statusline 명령으로 원하는 내용을 표시하도록 직접 꾸밀 수 있습니다.

실습
  1. Claude Code에서 /statusline 입력
  2. 원하는 내용을 자연어로 요청

    예) “모델 이름이랑 컨텍스트 사용량을 프로그레스 바로 보여줘”

  3. Claude가 ~/.claude/statusline-command.sh 스크립트와 ~/.claude/settings.json을 자동으로 설정해 줌

설정이 끝나면 상태줄이 이렇게 표시됩니다:

Claude Opus 4.8  ███████████░░░░░░░░░  55%
💡
표시할 수 있는 값은 상태줄이 실제로 제공하는 데이터로 한정됩니다. “실행 중인 작업 진행률 %” 같은 값은 제공되지 않아 만들 수 없고, 모델·컨텍스트 사용량·사용 한도 등은 표시할 수 있습니다.
⚠️
스크립트가 jq를 사용한다면 brew install jq로 먼저 설치해야 상태줄이 제대로 표시됩니다. (Mac 기준)
확인
  • 창 하단 상태줄에 모델 이름이 보임
  • 컨텍스트 사용량 프로그레스 바가 표시됨

Effort — Claude의 ‘생각 깊이’ 다이얼

/effort로 한 세션 동안 얼마나 깊게 생각할지 조절

이론

Effort(에포트)는 Claude가 답하기 전에 얼마나 깊게 생각(reasoning)할지를 정하는 세션 단위 설정입니다. 높일수록 더 꼼꼼히 생각해 품질이 올라가지만 토큰을 더 쓰고 느려지며, 낮추면 빠르고 저렴합니다.

레벨느낌언제
low빠르고 저렴간단한 수정·질문
medium가벼운 균형비용이 신경 쓰일 때
high (기본)품질·속도 균형대부분의 작업
xhigh더 깊게복잡한 설계·디버깅
max가장 깊게 (토큰 많이)정말 어려운 문제 하나

기본값은 대부분의 모델에서 high입니다. (일부 모델은 조금 다를 수 있어요.)

실습
  1. /effort만 입력 → 슬라이더로 레벨 선택
  2. 또는 바로 지정: /effort low · /effort xhigh
  3. /model로 모델을 고를 때 좌우 화살표로 effort도 함께 조절

    모델(어떤 두뇌를 쓸지)과 effort(얼마나 깊게)는 세트로 생각하면 편해요.

  4. 앞으로 계속 쓸 기본값은 /config에서 지정
💡
대화식으로 /effort low~xhigh를 설정하면 다음 세션에도 유지됩니다. 단 max는 그 세션에만 적용되고 저장되지 않아요.
⚡
/fast를 켜면 같은 Opus 모델을 더 빠른 출력으로 씁니다. effort(생각 깊이)와는 다른 축이라, 빠르게 받아보고 싶을 때 함께 활용하세요.
확인
  • /effort로 레벨을 바꿔봤다
  • 기본값이 high라는 걸 안다

워크플로우 모드 — 승인 프롬프트 조절

Shift+Tab으로 ‘얼마나 물어볼지’를 순환 전환

이론

Claude Code는 파일을 고치거나 명령을 실행하기 전에 “해도 될까요?”라고 물어봅니다. 이 승인 정도를 모드로 바꿀 수 있어요. 입력창에서 Shift+Tab을 누르면 모드가 순환합니다.

모드자동 승인 범위언제
default (기본)읽기만 자동, 수정·실행은 매번 확인안전하게 하나씩 확인
accept edits파일 수정까지 자동 승인믿고 맡기며 빠르게
plan읽기만 — 아무것도 바꾸지 않고 계획만먼저 방향부터 잡을 때

기본은 default이고, Shift+Tab → default → accept edits → plan 순으로 돌아요. (auto 등 고급 모드는 별도 설정 시 순환에 추가됩니다.)

실습
  1. 입력창에서 Shift+Tab을 몇 번 눌러 모드가 바뀌는 걸 확인
  2. 입력창 근처에 현재 모드가 표시되는지 보기
  3. plan 모드로 “이 기능 어떻게 만들지 계획만 세워줘” 시켜보기 (파일은 안 바뀜)
⚠️
bypass permissions(전부 묻지 않음) 같은 모드는 확인 없이 실행하므로, 격리된 환경이 아니라면 쓰지 마세요.
확인
  • Shift+Tab으로 모드를 바꿔봤다
  • plan / accept edits가 각각 뭘 하는지 안다
✦

딥띵킹 & ultrathink — ‘생각’ 켜고 끄기

Effort와 헷갈리기 쉬운 사고 과정 제어

이론

딥띵킹(Extended Thinking)은 Claude가 답하기 전에 하는 사고 과정입니다. Alt+T(맥은 Option+T)로 켜고 끌 수 있어요. Effort가 “얼마나 깊게”라면 딥띵킹 토글은 “사고 과정을 쓸지·보여줄지”를 다룹니다 — 둘은 별개예요.

💡
딥띵킹의 기본 상태(켜짐/꺼짐)는 공식 문서에 딱 정해져 있지 않습니다. 지금 켜져 있는지와 앞으로의 기본값은 /config에서 확인·변경할 수 있어요.

ultrathink — 이번 한 번만 더 깊게

프롬프트 아무 곳에나 ultrathink라는 단어를 넣으면 그 턴에만 더 깊이 생각하도록 요청합니다. (세션 effort 설정은 그대로예요.)

이 버그의 근본 원인을 ultrathink 로 찾아줘
ℹ️
현재 인식되는 사고 키워드는 ultrathink 하나입니다. think · think hard · think more 같은 단어는 특별한 효과 없이 일반 텍스트로 처리돼요. 평소의 생각 깊이는 ultrathink보다 /effort로 조절하는 게 기본입니다.

세 가지 한눈에

무엇조작한 줄 요약
/effort세션 전역평소 얼마나 깊게 생각할지
Alt+T세션 토글딥띵킹을 켤지 끌지
ultrathink프롬프트에 추가이번 턴만 더 깊게
확인
  • Alt+T(Option+T)로 딥띵킹을 토글해봤다
  • effort / 딥띵킹 / ultrathink가 각각 뭘 하는지 구분한다
  • 인식되는 키워드는 ultrathink뿐임을 안다

멀티인스턴스 & Git Worktree

동시에 여러 작업을 진행하는 두 가지 방법

이론

멀티 인스턴스 운영

탭 1: "Feature-Auth" — 인증 기능 개발
탭 2: "Bug-Fix"      — 버그 수정
탭 3: "Refactor"     — 리팩토링

터미널 탭 여러 개를 열고 각 탭에서 다른 Claude 인스턴스를 돌리세요. 탭 이름을 바꿔두면 한눈에 관리 가능합니다.

멀티 인스턴스
사람이 탭을 직접 스위칭
각각 완전히 독립된 세션
Sub-Agent (Day 2에서 배운 것)
Claude가 알아서 위임
한 세션 안에서 자동 실행

Git Worktree — 병렬 브랜치 작업

하나의 Git 저장소에서 여러 개의 작업 디렉토리를 만드는 기능. 커밋 히스토리는 공유하면서 파일 시스템은 완전히 분리됩니다.

# Claude Code 네이티브 지원
claude --worktree feature-auth
claude -w feature-auth
💡
claude -w 한 줄이면 워크트리 생성 → 브랜치 체크아웃 → 세션 시작까지 자동. 변경 없으면 세션 종료 시 자동 정리.

심화 — 워크트리 여러 개 동시 운영

claude -w feature-auth      # 워크트리 1: 인증 기능
claude -w bugfix-navbar     # 워크트리 2: 네비 버그 수정 (동시에!)

# 현재 워크트리 목록 확인
git worktree list

# 다 쓴 워크트리 정리
git worktree remove ../feature-auth

터미널 탭마다 서로 다른 워크트리에서 Claude를 띄우면, 완전히 독립된 파일 시스템에서 진짜 병렬로 작업할 수 있습니다 — 브랜치 전환 없이, 컨텍스트 섞임 없이.

⚠️
워크트리마다 npm install 등 의존성 설치가 따로 필요할 수 있습니다. 무거운 프로젝트라면 워크트리 개수를 2-3개로 제한하세요.
실습
  1. 터미널 탭 2개를 열고 각각 다른 작업을 지시해보기
  2. claude -w bugfix-navbar 실행해서 워크트리 체험해보기
  3. git worktree list로 목록 확인 → 다 쓴 워크트리 git worktree remove로 정리
확인
  • 멀티 인스턴스와 Sub-Agent의 차이를 안다
  • 상황에 따라 어떤 걸 쓸지 판단할 수 있다
  • Worktree를 하나 만들고 정리까지 해봤다
  • 워크트리 여러 개를 동시에 운영하는 법을 안다

종합 실습 & 데모데이 준비

3일간 배운 모든 걸 하나로 모아서 발표까지

이론

오늘의 대부분은 자유 작업 시간입니다. 지금까지 배운 모든 도구(Plan Mode, Skills, 슬래시 커맨드, Sub-Agent, MCP, Hooks, Git, Vercel, Worktree)를 자유롭게 조합해서 내 프로젝트를 마무리하고, 마지막에 다 함께 발표합니다.

실습

1부 — 자유 작업 (약 90분)

  1. Plan Mode로 오늘 마무리할 기능 목록 정리
  2. 작은 단위로 쪼개서 하나씩 완성 → 확인 → /clear
  3. 디자인이 아쉬우면 frontend-design 등 Skill로 다듬기
  4. code-reviewer Sub-Agent로 전체 코드 리뷰 한 번 받기
  5. (선택) Hook으로 "배포 완료" 알림 설정

2부 — 배포 & 최종 점검 (약 30분)

  1. 최종 커밋 정리 후 /commit → git push
  2. Vercel 배포가 최신 상태인지 확인 (자동 재배포 확인)
  3. 핸드폰 등 다른 기기에서 실제 배포 URL 접속해보기

3부 — 데모데이 (약 30분)

  1. 1분 피치 준비

    무엇을 만들었나 / 무엇을 배웠나 / 가장 재미있었던 순간, 세 가지로 정리.

  2. 배포 URL을 화면에 띄우고 순서대로 발표
  3. 서로 피드백 주고받기
💡
시간이 빠듯하면 1부를 줄이고 2부·3부(배포 확인 + 데모데이)는 꼭 지키세요. 데모데이가 오늘의 하이라이트입니다.
확인
  • 최신 코드가 Vercel에 배포되어 있다
  • Sub-Agent 리뷰를 받아봤다
  • 1분 피치가 준비됐다
  • 공유할 배포 URL을 확보했다
  • 발표를 마쳤다 🎉
