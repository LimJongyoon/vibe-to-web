## Part A — 작업 원칙 ⏱ 45분

### Step 8. 한 세션 = 한 피처 원칙

**[이론]**

Claude는 모든 대화를 머릿속에 들고 다닙니다. 대화가 길어질수록 더 무거워지고 결국 헷갈리기 시작합니다. **신선한 컨텍스트가 부풀어진 컨텍스트보다 항상 낫습니다.**

| ❌ 자주 하는 실수 | ✅ 올바른 워크플로우 |
|----------------|------------------|
| 한 세션에 여러 기능을 계속 추가 → 대화가 길어져 Claude가 헷갈림 | 작업 1 완료 → `/clear` → 작업 2 시작 → `/clear` → 반복 |

💡 `/clear`는 파일을 지우지 않습니다. **대화 기억만 지우고, 파일은 그대로 남습니다.**

**[실습]**

```
[작업 1] 자기소개 페이지 만들기 → 결과 확인 → /clear ✨
[작업 2] 취미 갤러리 추가 → 결과 확인 → /clear ✨
```

토큰 사용량 확인:
```
> /context
```
80% 이상이면 정리 타이밍.

| 명령어 | 차이점 | 언제 |
|--------|--------|------|
| `/clear` | 완전 초기화 | 새 작업 시작 시 (가장 많이 씀) |
| `/compact` | 요약 압축 (맥락 유지) | 같은 작업 계속하되 공간 확보 |

**[확인]**
- [ ] "한 세션 = 한 피처" 원칙을 외웠다
- [ ] `/clear`가 파일을 안 지운다는 걸 안다
- [ ] `/context`로 토큰 사용량 확인했다

### Step 9. Plan Mode를 습관처럼

**[이론]**

| 모드 | 의미 |
|------|------|
| **Plan Mode** | 계획만 세움 (실행 X) — 잘못된 방향을 30초에 잡을 수 있음 |
| **Accept Mode** | 계획 + 실행까지 함 (기본) |

Plan Mode가 필요한 경우:
- 새 기능을 통째로 추가할 때
- 여러 파일을 동시에 건드리는 작업
- 30분 이상 걸릴 것 같은 작업

**[실습]**

1. `Shift + Tab` → Plan Mode 진입
2. 큰 작업 설명:
   ```
   > 내 자기소개 페이지에 다크/라이트 모드 토글 기능을 추가하고 싶어.
     오른쪽 상단에 버튼 하나 두고, 클릭하면 전체 색상이 부드럽게 전환.
   ```
3. Claude의 계획을 받은 후 검토
4. 마음에 들면: `Shift + Tab` → Accept Mode → "좋아 진행해줘"
5. 수정 필요하면: "localStorage는 빼고 body 클래스만으로 가자."

💡 습관 팁: 새 채팅 시작할 때 첫 명령어로 `Shift+Tab`. 큰 작업이면 그대로, 작은 거면 다시 눌러서 Accept Mode로.

**[확인]**
- [ ] Plan Mode와 Accept Mode 전환 가능
- [ ] 실제로 Plan을 받아보고 계획을 읽고 수정 요청해봤다

## Part D — 버전관리 & 확장 도구 ⏱ 90분

### Step 14. Git에 올리기

**[왜 하는지]**

지금까지 만든 파일은 내 컴퓨터에만 있습니다. 컴퓨터가 고장 나거나 실수로 파일을 지우면 다 사라집니다. **Git**은 "이 시점의 파일 상태를 저장해두는 기능"이고, **GitHub**은 그렇게 저장한 걸 인터넷에 보관해주는 사이트입니다.

비유하자면, Git으로 저장하는 행위(커밋)는 게임의 "세이브 포인트"를 만드는 것과 같습니다. 나중에 뭔가 잘못돼도 그 세이브 포인트로 돌아갈 수 있습니다. GitHub은 그 세이브 파일을 클라우드에 백업해두는 것입니다.

앞으로 나올 용어 3개만 먼저 알아두면 됩니다.

| 용어 | 뜻 |
|---|---|
| 저장소(repository) | 내 프로젝트 폴더를 Git이 추적하는 상태로 만든 것 |
| 커밋(commit) | 지금 이 순간의 파일 상태를 저장하는 것 (세이브 포인트) |
| 푸시(push) | 내 컴퓨터에 저장한 커밋을 GitHub(인터넷)로 올려보내는 것 |

💡 오늘은 GitHub에 코드를 안전하게 올려두는 것까지만 합니다. 실제로 인터넷에 **배포(Vercel)**하는 건 내일 Day 3에서 합니다.

**[그대로 따라하기]**

1. **GitHub 계정 만들기**
   1. 브라우저에서 github.com 접속
   2. 오른쪽 위 Sign up 클릭
   3. 이메일, 비밀번호, 아이디 입력 (아이디는 나중에 `github.com/내아이디`로 쓰이니 신중하게)
   4. 이메일 인증 → 무료 플랜 선택
2. **GitHub Desktop 설치** (터미널 명령어 없이 마우스로만 진행)
   1. desktop.github.com 접속 → 다운로드 → 설치
   2. 실행 후 GitHub 계정으로 로그인
3. **내 프로젝트 폴더를 저장소로 등록**
   1. GitHub Desktop에서 `File → Add Local Repository`
   2. 지금 작업 중인 폴더(`my-first-web` 등) 선택
   3. "이 폴더는 아직 Git 저장소가 아닙니다"라는 메시지가 뜨면 `Create a Repository` 클릭
4. **첫 커밋 만들기**
   1. 왼쪽에 변경된 파일 목록이 보입니다
   2. 아래 Summary 칸에 `작업 완성` 같은 짧은 설명 입력
   3. `Commit to main` 클릭 → 이게 세이브 포인트 1개가 만들어진 것입니다
5. **GitHub에 올리기(푸시)**
   1. 위쪽 `Publish repository` 클릭
   2. "Keep this code private" 체크 **해제** (공개로 올려야 나중에 배포 링크 공유 가능)
   3. `Publish` 클릭
6. **확인** — 브라우저에서 `github.com/내아이디` 접속 → 방금 올린 저장소가 보이면 성공입니다.

**이후 수정할 때마다**: 파일을 고친 뒤 GitHub Desktop을 열면 왼쪽에 바뀐 부분이 자동으로 표시됩니다. Summary 입력 → `Commit to main` → 위쪽 `Push origin` 버튼만 누르면 됩니다. 이 세 단계(수정 → 커밋 → 푸시)가 앞으로 계속 반복되는 전체 흐름입니다.

⚠️ 비밀번호나 API 키가 들어있는 파일은 절대 올리지 마세요. Claude에게 ".gitignore 파일 만들어줘, 비밀번호 파일은 Git에 안 올라가게 해줘"라고 요청하면 자동으로 처리해줍니다.

**[확인]**
- [ ] GitHub 계정을 만들었다
- [ ] GitHub Desktop으로 내 폴더를 저장소로 등록했다
- [ ] 커밋 → 푸시를 한 번 해봤다
- [ ] 브라우저에서 내 코드가 GitHub에 올라간 걸 확인했다

<details>
<summary>📎 선택 심화 — 터미널로 하는 방법 (마우스 방식이 익숙해진 뒤에)</summary>

```bash
# 최초 한 번만 설정
git config --global user.name "본인이름"

git commit -m "수정 내용 설명"
```

### Step 15. Skills & 슬래시 커맨드 — 반복 작업 자동화

**[왜 하는지]**

같은 요청을 매번 길게 타이핑하고 계신가요? 예를 들어 "다크모드로, Pretendard 폰트로, 반응형으로 만들어줘"를 새 페이지 만들 때마다 반복 입력한다면, 그걸 파일 하나로 저장해두고 재사용하는 기능이 Skills와 슬래시 커맨드입니다.

둘의 차이는 딱 하나입니다.

| | Skill | 슬래시 커맨드 |
|---|---|---|
| 실행 방법 | Claude가 상황 보고 **알아서** 실행 | 내가 `/이름`을 **직접** 입력해야 실행 |
| 비유 | 상황 되면 자동으로 켜지는 센서등 | 내가 눌러야 켜지는 스위치 |

**[그대로 따라하기]**

**슬래시 커맨드부터 (더 쉬운 쪽):**
1. **폴더와 파일 만들기** — 프로젝트 폴더 안에 `.claude/commands/commit.md` 파일을 만듭니다. (Claude에게 "`.claude/commands/commit.md` 파일 만들어줘"라고 시켜도 됩니다.)
2. **내용 채우기**:
   ```
   1. git status/git diff로 변경사항 확인
   2. 변경 내용을 요약한 한국어 커밋 메시지 초안 작성
   3. 사용자에게 확인 요청
   4. 승인되면 git add -A && git commit -m ... 실행
   ```
3. **실행해보기**:
   ```
   > /commit
   ```
   입력하면 Claude가 위 4단계를 순서대로 실행합니다. 방금 14단계에서 마우스로 하던 커밋 작업을 이제 이 한 줄로 대체할 수 있습니다.

**이어서 Skill (자동 트리거):**
1. **폴더와 파일 만들기** — `.claude/skills/새-페이지-만들기/SKILL.md` 파일을 만듭니다.
2. **맨 위에 이렇게 채우기** (이 형식이 고정입니다):
   ```yaml
   ---
   name: 새-페이지-만들기
   ```

### Step 16. Sub-Agent — 도우미 하나 더 부르기

**[왜 하는지]**

지금까지는 Claude 하나와만 대화했습니다. Sub-Agent는 "이 작업만 전담하는 도우미를 하나 더 불러서 시키는 것"입니다. 예를 들어 코드 검토처럼 결과물이 길게 나오는 작업을 Sub-Agent에게 맡기면, 그 긴 결과가 메인 대화창을 어지럽히지 않고 요약만 돌아옵니다.

처음에는 "이미 만들어져 있는 Sub-Agent를 불러 쓰는 법"만 알면 충분합니다. 직접 만드는 건 선택 심화로 뒤에 있습니다.

**[그대로 따라하기]**

그냥 자연어로 요청하기 (가장 쉬움):
```
> 서브에이전트로 최근 변경사항 코드 리뷰해줘
```
"서브에이전트로"라는 말만 넣어도 충분합니다. 어떤 도우미를 쓸지는 Claude가 알아서 고릅니다.

기본으로 내장된 도우미 5종 (이미 있으니 만들 필요 없음):

| 이름 | 용도 |
|------|------|
| Explore | 코드/파일 탐색, 구조 파악 |
| Plan | Plan Mode에서 계획 세울 때 조사 |
| General-purpose | 탐색+수정이 같이 필요한 복잡한 작업 |
| Bash | 터미널 명령만 따로 실행 |
| Claude Code Guide | Claude Code 사용법 질문 답변 |

확인해보기:
```
> 서브에이전트로 이 프로젝트 폴더 구조 파악해서 알려줘
```
실행하면 잠깐 별도로 작업하는 표시가 뜨고, 끝나면 요약된 결과만 메인 대화에 나타납니다. 그게 Sub-Agent가 작동한 것입니다.

**[확인]**
- [ ] "서브에이전트로 ~해줘" 형태로 한 번 요청해봤다
- [ ] 결과가 요약되어서 돌아오는 걸 확인했다

<details>
<summary>📎 선택 심화 — 내 전용 Sub-Agent 직접 만들기</summary>

직접 파일을 쓰는 대신, 그냥 Claude에게 만들어달라고 요청하는 게 제일 쉽습니다.

```
> code-reviewer라는 서브에이전트를 만들어줘.
  코드 품질/보안 검토 역할,
  Read/Grep/Glob/Bash 도구만 쓰게 해줘.
```

Claude가 `.claude/agents/code-reviewer.md` 파일을 아래와 같은 형식으로 자동 생성해줍니다. 직접 쓸 경우 형식은 이렇습니다.

```yaml
---
name: code-reviewer
description: "코드 리뷰 전문가. 품질, 보안, 모범 사례를 검토."
tools: Read, Grep, Glob, Bash
model: sonnet
---

호출되면:
1. git diff로 최근 변경 확인
2. 수정된 파일에 집중하여 리뷰
```

만든 뒤에는 `@code-reviewer`처럼 이름 앞에 `@`을 붙여서 확실하게 지정 호출할 수도 있습니다.

⚠️ 예전 버전에서는 `/agents`로 마법사 UI가 떴지만, 최신 버전에서는 파일 기반으로만 만듭니다. `/agents`는 지금 실행 중인 서브에이전트를 보는 모니터링 화면입니다.
</details>

### Step 17. MCP — 외부 서비스와 연결하기

**[왜 하는지]**

Claude가 내 컴퓨터 파일뿐 아니라 Slack, Notion 같은 외부 서비스도 다룰 수 있게 해주는 연결 방식입니다. "코드를 직접 짜지 않아도 되는 플러그인을 꽂는 것"이라고 생각하면 됩니다.

오늘은 **이미 존재하는 서버를 연결해서 써보는 것**까지만 합니다. 커스텀 서버를 직접 만드는 건 나중에 필요할 때 해도 늦지 않습니다.

**[그대로 따라하기]**

1. **연결하기**:
   ```bash
   # 내 컴퓨터에서 실행되는 도구를 연결할 때
   claude mcp add <이름> -- npx -y <패키지이름>

   # 인터넷 너머 서버에 연결할 때
   claude mcp add --transport http <이름> <서버주소>
   ```
   `<이름>`, `<패키지이름>`, `<서버주소>`는 실제 값으로 바꿔서 입력합니다. 예를 들어 파일시스템 도구를 연결한다면:
   ```bash
   claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem
   ```
2. **잘 연결됐는지 확인**:
   ```bash
   claude mcp list
   ```
   방금 연결한 이름이 목록에 뜨면 성공입니다.
3. **사용해보기** — 연결만 하면 별도로 부르지 않아도, 관련된 요청을 할 때 Claude가 알아서 그 도구를 씁니다.

**[확인]**
- [ ] `claude mcp add`로 서버 하나를 연결해봤다
- [ ] `claude mcp list`로 연결 상태를 확인했다

<details>
<summary>📎 선택 심화 — MCP 대신 간단한 Bash 스크립트로 충분한 경우</summary>

작업이 단순하면 MCP 연결 없이 로컬 스크립트만으로 충분합니다. 연결 과정도 없고 대화창 공간도 아낄 수 있습니다.

```bash

psql -h localhost -U dev -d mydb -c "SELECT count(*) FROM users;"

```


MCP는 "매 대화마다 도구 설명이 자동으로 포함되는" 방식이라, 정말 반복적으로 외부 서비스와 연동해야 할 때만 쓰는 게 맞습니다. 간단한 일회성 작업은 그냥 Bash가 더 가볍습니다.

커스텀 MCP 서버가 필요해지면, 이렇게 요청만 하면 Claude가 코드 작성부터 등록까지 도와줍니다.

### Step 18. Hooks — 자동화 엔진

**[왜 하는지]**

"파일을 저장할 때마다 자동으로 정리해줘" 같은, **특정 상황이 되면 자동으로 실행되는 규칙**입니다. 오늘은 가장 흔히 쓰는 예시 하나(작업 끝나면 알림 보내기)만 직접 만들어봅니다.

**[그대로 따라하기]**

```
> 알림 Hook 만들어줘.
  작업이 끝나면 소리랑 알림 띄워줘.
```
이렇게 요청하면 Claude가 알아서 설정 파일(`settings.json`)에 필요한 내용을 추가해줍니다. 직접 파일을 열어서 수정할 필요는 없습니다.

실행해서 확인하기: 아무 작업이나 하나 시켜본 뒤, 끝났을 때 알림이 뜨는지 확인합니다.

⚠️ Hook이 실행되는 동안 Claude는 멈춰서 기다립니다. 만든 뒤 반응이 없다면 "Hook이 멈춰있는 것 같아, timeout 설정해줘"라고 요청하세요.

**[확인]**
- [ ] 알림 Hook을 만들어봤다
- [ ] 작업이 끝났을 때 알림이 뜨는 걸 확인했다

<details>
<summary>📎 선택 심화 — Hook이 반응하는 4가지 시점</summary>

지금 만든 알림 Hook은 아래 표에서 `Stop` 시점에 해당합니다. 필요할 때 아래 표를 참고해서 다른 시점의 Hook도 만들 수 있습니다.

| 시점 | 언제 | 예시 |
|------|------|------|
| PreToolUse | 도구를 실행하기 직전 | 위험한 명령어 실행 전에 승인 요청 |
| PostToolUse | 도구 실행 직후 | 코드 저장 직후 자동 포맷팅 |
| Notification | 사용자 응답을 기다릴 때 | 알림 전송 |
| Stop | Claude의 턴이 끝났을 때 | 완료 알림, 최종 정리 |
</details>
